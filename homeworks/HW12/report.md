# HW12 – временные ряды: temporal split, baseline-модели и GRU-прогноз

## 1. Кратко: что сделано

- **Датасет и задача.** Использовался датасет `S12-hw-dataset.csv`, прогнозировалось значение `target` с горизонтом 1 час.
- **Разбиение.** Хронологическое разделение без перемешивания: первые 65% — train, следующие 17.5% — validation, последние 17.5% — test.
- **Сравниваемые подходы.** B1 — наивный прогноз последним значением (`y_t = y_{t-1}`). B2 — скользящее среднее с окном 7 часов. B3 — Ridge-регрессия на лаговых, rolling- и календарных признаках. R1 — однослойная GRU (hidden_size=64) на нормализованных последовательностях длиной 36 шагов.
- **Лучший подход.** GRU (`R1`) показала наименьший `val_MAE=5.11` и `test_MAE=5.49`. Способность моделировать нелинейные зависимости и долгосрочный контекст обеспечила преимущество над линейными методами.

## 3. Данные и постановка задачи

- Датасет: `S12-hw-dataset.csv`
- Целевая колонка: `target`
- Частота временного ряда: 1 час
- Горизонт прогноза: 1 шаг (1 час)
- Размер датасета: 4320 наблюдений
- Диапазон дат: 2025-01-01 00:00:00 — 2025-06-29 23:00:00
- Дополнительные внешние признаки: отсутствуют
- **Комментарий.** Ряд характеризуется устойчивым восходящим трендом на протяжении всего периода и чётко выраженной суточной периодичностью. Пропуски отсутствуют, аномальные значения не критичны, однако к завершающему этапу наблюдается увеличение волатильности. Несмотря на сложность, задача прогноза на один час решаема даже простыми алгоритмами, но применение нелинейных архитектур даёт ощутимый прирост точности.

## 4. Temporal split и признаки

### 4.1. Разбиение по времени

- **Как выполнен split.** Хронологическое разбиение без перемешивания: первые 65% (2808 точек) — train, следующие 17.5% (756 точек) — validation, последние 17.5% (756 точек) — test. Все модели используют одинаковые граничные даты.
- **Почему `random split` здесь некорректен.** Временные ряды обладают автокорреляцией и трендом. Случайное перемешивание приводит к утечке информации из будущего в обучающую выборку, завышая метрики и искажая оценку реальной прогностической способности модели.

### 4.2. Признаки для baseline-моделей

- **Лаги:** lag_1, lag_7, lag_14.
- **Rolling-признаки:** скользящее среднее (`rolling_mean_7`) и стандартное отклонение (`rolling_std_7`) с окном 7, вычисленные через `shift(1)` для исключения утечки.
- **Календарные признаки:** час суток, день недели, месяц.
- **Обработка пропусков после лагов/rolling:** первые 14 строк (ограничение lag_14) удалялись через `dropna()`.
- **Масштабирование:** для B1 и B2 не применялось. Для B3 — `StandardScaler`, обученный только на train. Для GRU — `StandardScaler` на исходном ряду, также `fit` только на train.

## 5. Модели и эксперименты (B1, B2, B3, R1)

- **B1 (`naive-last`):** прогноз равен последнему известному значению (`y_t = y_{t-1}`). Обучение отсутствует. `window_size=N/A`.
- **B2 (`moving-average`):** скользящее среднее с окном 7: `y_t = mean(y_{t-7}...y_{t-1})`. Обучение отсутствует. `window_size=7`.
- **B3 (`ridge-lag-features`):** Ridge-регрессия на инженерных признаках (лаги, rolling-статистики, временные фичи). `alpha=1.0`. `window_size=N/A` (Ridge не использует оконное представление).
- **R1 (`gru-forecast`):** однослойная GRU на масштабированных последовательностях. `hidden_size=64`, `window_size=36`, `horizon=1`.

Дополнительно:

- Основная метрика выбора лучшей модели: `val_MAE`
- Дополнительные метрики: `val_RMSE`, `val_MAPE`, `test_RMSE`, `test_MAPE`
- `window_size` для GRU: 36
- Batch size: 96
- Максимальное число эпох: 20
- Optimizer: Adam
- Learning rate: 2e-3
- Как сохранялась лучшая модель: при каждом улучшении `val_loss` веса сохранялись в `artifacts/best_gru.pt`, конфигурация — в `artifacts/best_gru_config.json`.

## 7. Анализ

Наивный прогноз последним значением (B1) оказался неожиданно конкурентоспособным (`test_MAE=6.33`), что объясняется высокой автокорреляцией в почасовом ряду: соседние наблюдения действительно сильно связаны. Однако скользящее среднее с окном 7 (B2) значительно ухудшило результат (`test_MAE=12.84`): сглаживание «размывает» локальные колебания и не успевает адаптироваться к возрастающей волатильности во второй половине периода.

Введение лагов, скользящих статистик и временных признаков в линейную модель Ridge (B3) дало существенный скачок качества (`test_MAE=5.76`). Это подтверждает, что даже простая линейная модель способна эффективно улавливать структуру ряда при условии подачи релевантных характеристик. Все признаки строились с `shift(1)` перед rolling, что исключает утечку информации из будущего.

GRU продемонстрировала наивысшую точность, обойдя все эталонные подходы. Её преимущество заключается в способности моделировать нелинейные зависимости и сохранять долгосрочный контекст окна 36 шагов. Тем не менее прирост относительно Ridge составил около 12% по MAE на валидации, что указывает на высокую ценность правильно сконструированных признаков в данной задаче.

В данных отчётливо прослеживаются восходящий тренд и суточная периодичность. Модели, учитывающие эти закономерности через лаги и календарные признаки, закономерно обходят инерционные методы.

Потенциальные источники утечки данных в подобных задачах: случайное (а не временное) разбиение, вычисление rolling-статистик без `shift(1)`, масштабирование на всём массиве до разделения. В работе все эти риски устранены: split — по датам, rolling — через `shift(1)`, `StandardScaler` обучается только на train.

Наиболее ощутимые ошибки всех моделей наблюдались в моменты резких изменений целевой переменной. Инерционные алгоритмы отставали в таких точках; GRU адаптировалась быстрее за счёт контекста окна, но и она не всегда успевала при внезапных скачках.

## 8. Итоговый вывод

В качестве базового решения для подобных задач рекомендуется линейная модель с конструированными признаками (лаги, скользящие статистики, временные характеристики) — она обеспечивает хорошее качество при низкой вычислительной сложности и интерпретируемости. Ограничиться Ridge достаточно, если ряд характеризуется высокой автокорреляцией, требования к точности умеренны, а ресурсы ограничены.

Переход к GRU оправдан при необходимости максимальной точности, наличии достаточного объёма данных и вычислительных ресурсов. В данном эксперименте GRU улучшила `val_MAE` примерно на 12% относительно Ridge, что является значимым, но умеренным приростом.

Главный вывод о валидации: хронологический порядок нельзя нарушать. Обучение — только на прошлых данных, валидация — на ближайшем будущем, тестирование — на отдалённом отрезке, один раз. Только такой подход обеспечивает реалистичную оценку прогностической способности модели.
