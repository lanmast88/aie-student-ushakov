## 1. Паспорт проекта

- **Название проекта:** Сервис рекомендаций похожих туров (Similar Tours Recommender)
- **Автор:** Ушаков Глеб Максимович
- **Группа:** ЭФБО-06-24

- **Ссылка на репозиторий:** [github.com/lanmast88/aie-student-ushakov](https://github.com/lanmast88/aie-student-ushakov.git)

Проект посвящён построению сервиса рекомендаций похожих туров на основе content-based подхода.
Используется синтетический датасет из 200 туров по 10 городам; модель комбинирует структурные признаки (город, категория, звёздность, цена, длительность, тип питания) и TF-IDF-представление текстовых описаний.
Результат — REST API, которое по идентификатору тура возвращает Top-K наиболее похожих туров из каталога.

## 2. Постановка задачи и контекст

### 2.1. Предметная область и задача

Туристические платформы нуждаются в механизме подбора похожих предложений: пользователь просматривает конкретный тур и хочет видеть аналоги по направлению, классу и бюджету.
Тип задачи: **content-based retrieval** — поиск ближайших соседей в пространстве признаков тура без истории взаимодействий пользователя.
Целевая аудитория: посетители тревел-платформы, просматривающие карточку тура и нуждающиеся в альтернативах.

### 2.2. Формулировка задачи в терминах ML/ИИ

- **Вход:** `tour_id` (уникальный идентификатор тура) и опциональный параметр `top_k` (целое, 1–20, по умолчанию 5).
- **Выход:** упорядоченный список из `top_k` туров с полями `id`, `name`, `city_name`, `category`, `hotel_stars`, `meal_type`, `price`, `similarity` (cosine similarity).
- **Ограничения и требования:**
  - Cold-start не является проблемой — рекомендации content-based, на истории пользователя не опираются.
  - Задержка ответа должна быть на уровне единиц миллисекунд (вся матрица схожести хранится в памяти).
  - Решение должно быть воспроизводимым: матрица схожести вычислена заранее и сохранена как артефакт.

### 2.3. Целевые метрики качества

| Метрика | Описание | Почему выбрана |
|---|---|---|
| **CHR@5** (Category Hit Rate) | Доля рекомендаций той же категории (luxury / comfort / budget), что и запрос | Главный бизнес-сигнал: пользователю нужен тур схожего класса |
| **Coverage** | Доля туров каталога, которые хотя бы раз попадают в рекомендации | Защита от вырожденного решения («показывать одни и те же туры всем») |
| **MeanSim@5** | Средняя cosine similarity топ-5 рекомендаций | Мера плотности кластера: насколько найденные туры действительно близки |
| **ILD@5** (Intra-List Diversity) | Среднее попарное расстояние 1 − sim внутри списка | Разнообразие внутри выдачи: один список не должен состоять из идентичных туров |
| **Personalization@5** | 1 − среднее попарное пересечение списков для разных запросов | Разным запросам — разные ответы; защита от тривиального решения |

## 3. Данные

### 3.1. Источник данных

Синтетический датасет `data/tours.json`, сгенерированный скриптом с реалистичной бизнес-логикой:
цены коррелируют с городом, звёздностью отеля и сезоном; категории распределены неравномерно по городам (Мальдивы — только luxury; Барселона и Стамбул — преимущественно comfort).
Текстовые описания туров сгенерированы с помощью Claude (Anthropic).
Персональных и чувствительных данных нет.

### 3.2. Структура данных

**Файлы:**

| Файл | Описание |
|---|---|
| `data/tours.json` | Основной датасет — 200 туров |
| `artifacts/exp*/tours_active.json` | Отфильтрованная выборка активных туров (170), зафиксированная для экспериментов |
| `data/data_description.md` | Подробное описание всех полей |

**Ключевые признаки:**

| Признак | Тип | Значения / диапазон |
|---|---|---|
| `city_name` | категориальный | 10 городов (Bali, Barcelona, Cairo, Dubai, Istanbul, Maldives, New York, Paris, Prague, Tokyo) |
| `category` | категориальный | luxury (~50 %), comfort, budget |
| `meal_type` | категориальный | breakfast (~42 %), all_inclusive (~32 %), none (~27 %) |
| `hotel_stars` | числовой | 2–5 |
| `price` | числовой | €353 – €14 000, медиана €1 837, правосторонний скос |
| `duration_nights` | числовой | 3–14, пик на 10 ночах |
| `description` | текст | 2–3 предложения на тур |
| `status` | категориальный | active (170) / inactive (30) |

### 3.3. Предобработка и ключевые наблюдения из EDA

**Предобработка:**
- Фильтрация по `status == "active"` (170 из 200 туров).
- One-hot encoding категориальных признаков: `city_id`, `category`, `meal_type` → 16 бинарных колонок.
- Нормализация числовых признаков: `hotel_stars`, `duration_nights`, `log(price + 1)` → MinMaxScaler → 3 колонки.
- Итоговая структурная матрица: 170 × 19.
- TF-IDF по полю `description`: `max_features=200`, `ngram_range=(1, 2)` → матрица 170 × 200.

**Ключевые наблюдения из EDA** (`notebooks/exp01_eda.ipynb`):
- Распределение цен **правосторонне скошено**: используется `log(price + 1)` при нормализации.
- Мальдивы (медиана ~€9 587) и Дубай (~€7 137) — дорогие направления; Стамбул (~€1 059) и Прага (~€962) — доступные.
- `meal_type` несёт ценовой сигнал только **в связке с городом** — изолированно признак слабее.
- Корреляция цены и звёздности умеренная (r ≈ 0.54); с длительностью слабая (r ≈ 0.31).
- `start_date` не используется как признак: в content-based подходе без истории пользователя дата не несёт сигнала схожести.

## 4. Модели и подходы

### 4.1. Baseline — популярность в городе

**Описание:** для запрошенного тура возвращаются туры того же города, отсортированные по убыванию звёздности отеля. Логика: «если ищешь тур в Барселону — покажи лучшие отели Барселоны».

**Реализация:** `src/models/recommender.py` → `get_popularity_recs`.

**Слабые места:** Coverage = 0.40 — 60 % каталога никогда не попадает в рекомендации; тур из Дубая может никогда не быть рекомендован (если все его соседи по городу уже заняли топ-5).

### 4.2. Content-based: структурные признаки

**Описание:** матрица схожести вычисляется как попарная cosine similarity структурной матрицы (one-hot + нормированные числовые). Два тура схожи, если у них одинаковая категория, город, тип питания и близкие цена/звёздность/длительность.

**Реализация:** `src/models/features.py` → `build_feature_matrices`, `src/models/recommender.py` → `get_similar_tours`.

### 4.3. Content-based: TF-IDF по описаниям

**Описание:** матрица схожести вычисляется по cosine similarity TF-IDF-векторов описаний туров. Захватывает семантическую близость: туры в схожих направлениях («пляжный отдых», «деловой центр», «исторические достопримечательности») имеют высокую схожесть.

**Плюс:** Coverage = 1.0 — каждый тур хотя бы раз попадает в рекомендации.
**Минус:** CHR@5 = 0.725 — хуже структурного подхода; описания менее дискриминативны для категории (luxury / comfort / budget).

### 4.4. Комбинированная модель (финальная)

**Описание:** взвешенная сумма двух матриц:

```
sim_combined = w_struct · sim_struct + w_text · sim_tfidf
```

Объединяет точность структурных признаков с полнотой покрытия TF-IDF.

**Подбор весов:** grid search по `w_struct ∈ {0.1, 0.2, ..., 0.9}` в `notebooks/exp03_weight_tuning.ipynb`.
**Финальные веса:** `w_struct = 0.7`, `w_text = 0.3` (задано в `configs/config.yaml`).

## 5. Экспериментальный протокол и результаты

### 5.1. Экспериментальный протокол

- **Данные:** 170 активных туров из `data/tours.json`.
- **Разбивка:** train/val/test не требуется — задача unsupervised retrieval, нет целевых меток. Метрики считаются по всем 170 запросам (каждый тур является одновременно запросом и элементом каталога).
- **Оценка:** для каждого тура берётся Top-5 рекомендаций (исключая сам тур), затем усредняются CHR@5, Coverage, MeanSim@5, ILD@5 и Personalization@5.
- **Воспроизводимость:** артефакты (матрица схожести, TF-IDF векторайзер, scaler) сохранены в `artifacts/exp02/` и `artifacts/exp03/`.

### 5.2. Сравнение моделей — Эксперимент 2 (`notebooks/exp02_baseline.ipynb`)

| Модель / конфигурация | CHR@5 | Coverage | MeanSim@5 | ILD@5 | Pers@5 |
|---|---|---|---|---|---|
| Quality Baseline (популярность в городе) | 0.531 | 0.400 | 0.709 | 0.187 | 0.922 |
| Структурные признаки (only struct) | 0.934 | 0.982 | 0.878 | 0.176 | 0.969 |
| TF-IDF по описаниям (only text) | 0.725 | **1.000** | 0.427 | **0.707** | 0.970 |
| **Комбинированная (w=0.7)** | **0.936** | **1.000** | 0.720 | 0.355 | **0.972** |

### 5.3. Подбор весов — Эксперимент 3 (`notebooks/exp03_weight_tuning.ipynb`)

Grid search по `w_struct` от 0.1 до 0.9 с шагом 0.1 (детали — `artifacts/exp03/weight_search.csv`):

| Конфигурация | w_struct | CHR@5 | Coverage | MeanSim@5 | ILD@5 |
|---|---|---|---|---|---|
| w=0.1 (TF-IDF доминирует) | 0.1 | 0.786 | 0.994 | 0.457 | 0.670 |
| w=0.5 (равный вес) | 0.5 | 0.932 | 1.000 | 0.621 | 0.468 |
| **w=0.7 (оптимальный)** | **0.7** | **0.936** | **1.000** | **0.720** | **0.355** |
| w=0.9 (макс. CHR) | 0.9 | 0.940 | 0.994 | 0.824 | 0.241 |

### 5.4. Выбор финальной модели

Финальная конфигурация: **`w_struct = 0.7`, `w_text = 0.3`**.

**Обоснование:**
- Конфигурация `w=0.7` даёт Coverage = 1.000 — каждый тур хотя бы раз появляется в рекомендациях. При `w=0.9` Coverage опускается до 0.994, то есть несколько туров никогда не рекомендуются — неприемлемо для витрины.
- CHR@5 при `w=0.7` равен 0.936 — практически идентичен `w=0.9` (0.940), разница незначима.
- ILD@5 = 0.355 при `w=0.7` даёт умеренное разнообразие внутри списка: рекомендации не копируют запрос, но остаются тематически близкими.
- Trade-off: более высокий `w_text` улучшает ILD, но значительно снижает CHR (главный бизнес-сигнал).

## 6. Архитектура решения и сервис

### 6.1. Пайплайн

```
data/tours.json
    └── фильтрация active (170 туров)
            ├── build_feature_matrices()
            │       ├── one-hot: city_id, category, meal_type
            │       ├── MinMaxScaler: hotel_stars, duration_nights, log(price)
            │       └── TF-IDF: description (max_features=200, ngram=(1,2))
            ├── cosine_similarity(struct_matrix)  → sim_struct  [170×170]
            ├── cosine_similarity(tfidf_matrix)   → sim_tfidf   [170×170]
            └── 0.7·sim_struct + 0.3·sim_tfidf   → sim_combined [170×170]

Артефакты сохраняются в artifacts/exp03/:
    sim_combined.npy, scaler.pkl, tfidf_vectorizer.pkl, tours_active.json

FastAPI (src/service/app.py):
    @startup → load sim_combined.npy + tours_active.json в память
    GET /health       → статус сервиса
    GET /tours        → каталог (фильтры ?city, ?category)
    POST /predict     → get_similar_tours() → Top-K по sim_combined
```

### 6.2. API — эндпоинты

| Метод | Путь | Параметры | Описание |
|---|---|---|---|
| GET | `/health` | — | Статус сервиса, кол-во туров, форма матрицы |
| GET | `/tours` | `city` (opt), `category` (opt) | Список туров с фильтрацией |
| POST | `/predict` | `tour_id`, `top_k` (opt, 1–20) | Top-K похожих туров |

**Пример запроса:**
```json
POST /predict
{"tour_id": "362b3800-ccd0-47d2-b8cc-158bd0d22742", "top_k": 5}
```

**Пример ответа:**
```json
{
  "tour_id": "362b3800-ccd0-47d2-b8cc-158bd0d22742",
  "recommendations": [
    {"id": "...", "name": "Comfort Istanbul — 10 nights", "similarity": 0.981, ...}
  ],
  "elapsed_ms": 1.4
}
```

## 7. Наблюдаемость, конфигурация и безопасность

### 7.1. Логи и наблюдаемость

Используется стандартный модуль `logging` Python. Уровень логирования задаётся через `configs/config.yaml` (по умолчанию `INFO`).

Что логируется:
- **Startup:** путь к конфигу, директория артефактов, количество загруженных туров, форма матрицы, top_k по умолчанию.
- **Каждый запрос `/predict`:** `tour_id`, `top_k`, количество рекомендаций, время ответа в мс.
- **Ошибки:** `WARNING` при обращении к несуществующему `tour_id`.

Endpoint `/health` позволяет убедиться, что сервис поднялся и загрузил данные:
```json
{"status": "ok", "tours_loaded": 170, "sim_matrix_shape": [170, 170]}
```

### 7.2. Конфигурация

Все параметры вынесены в `configs/config.yaml`:

```yaml
artifacts_dir: artifacts/exp03    # какие артефакты загружает сервис
model:
  struct_weight: 0.7              # вес структурных признаков
  tfidf_weight: 0.3               # вес TF-IDF
  top_k: 5                        # Top-K по умолчанию
  tfidf_max_features: 200
  tfidf_ngram_range: [1, 2]
service:
  port: 8000
  log_level: INFO
```

Путь к конфигу можно переопределить через переменную окружения `CONFIG_PATH`.

### 7.3. Безопасность

- Репозиторий не содержит реальных секретов или API-ключей: сервис не обращается к внешним API.
- Данные полностью синтетические — персональных данных нет.
- Единственная переменная окружения — `CONFIG_PATH` — задокументирована в `configs/.env.example`; имеет безопасное значение по умолчанию, токенов и паролей не требует.
- Параметр `top_k` на сервисе ограничен диапазоном [1, 20] во избежание злоупотребления.

## 8. Ограничения и дальнейшая работа

### Текущие ограничения

- **Синтетические данные:** качество на реальном каталоге потребует переобучения и возможно смены весов.
- **Content-based only:** нет учёта поведения пользователей, популярности туров, истории покупок.
- **Матрица фиксирована:** при добавлении нового тура требуется пересчёт всей матрицы схожести и перезапуск сервиса.
- **Нет автоматических тестов:** папка `tests/` содержит только README.
- **`start_date` не используется:** в текущей версии нет фильтрации по доступности тура на конкретные даты.

### Направления для развития

- Добавить фильтрацию по дате отправления на уровне сервиса (`start_date`).
- Реализовать гибридную модель: content-based + popularity boost.
- Заменить статическую матрицу на приближённый поиск ближайших соседей (FAISS, Annoy) для масштабирования.
- Добавить модульные тесты (`pytest`) на функции `build_feature_matrices`, `get_similar_tours`, метрики.
- Добавить мониторинг запросов (Prometheus / встроенные метрики FastAPI).
- Реализовать инкрементальное обновление матрицы при добавлении нового тура.
