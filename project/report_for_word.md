## ВВЕДЕНИЕ

Туристическая индустрия активно переходит в онлайн-формат: сотни и тысячи предложений
публикуются на одной платформе, и перед пользователем стоит задача выбора из потока
однотипных карточек туров. Один из ключевых инструментов повышения конверсии и удержания
пользователя — блок «Похожие туры» на странице просматриваемого предложения.

Цель проекта — спроектировать и развернуть сервис рекомендаций похожих туров,
который по идентификатору тура возвращает Top-K наиболее близких альтернатив из каталога.
Решение реализовано в виде REST API на базе FastAPI и упаковано в Docker-контейнер.

Задачи проекта:

1) провести разведочный анализ синтетического датасета туров (EDA);
2) построить baseline и не менее двух улучшенных моделей с измерением качества по набору
   метрик;
3) подобрать оптимальную конфигурацию финальной модели методом перебора гиперпараметров;
4) реализовать воспроизводимый REST-сервис с эндпоинтами /health, /tours и /predict;
5) обеспечить логирование, конфигурируемость и безопасность сервиса.

Тип задачи — content-based retrieval: поиск ближайших соседей в пространстве признаков тура
без использования истории взаимодействий пользователя. Пользователь сервиса — посетитель
тревел-платформы, просматривающий карточку тура и нуждающийся в альтернативах.

## 1 ПОСТАНОВКА ЗАДАЧИ И АНАЛИЗ ДАННЫХ

### 1.1 Постановка задачи

Формальная постановка задачи в терминах машинного обучения:

- **Вход:** идентификатор тура (tour_id, строка UUID) и опциональный параметр top_k
  (целое число от 1 до 20, по умолчанию 5).
- **Выход:** упорядоченный список из top_k туров с полями id, name, city_name, category,
  hotel_stars, meal_type, price и значением косинусной схожести (similarity).
- **Ограничения:** задержка ответа на уровне единиц миллисекунд (матрица схожести хранится
  в оперативной памяти); холодный старт не является проблемой — рекомендации опираются
  исключительно на признаки тура, без истории пользователя.

Целевые метрики и обоснование их выбора приведены в таблице 1.1.

**Таблица 1.1 — Целевые метрики качества**

| Метрика | Описание | Обоснование |
|---|---|---|
| CHR@5 (Category Hit Rate) | Доля рекомендаций той же категории (luxury / comfort / budget), что и запрос | Главный бизнес-сигнал: пользователь ожидает тур схожего класса |
| Coverage | Доля туров каталога, попавших в рекомендации хотя бы раз | Защита от вырожденного решения — показывать одни и те же «хиты» |
| MeanSim@5 | Средняя косинусная схожесть топ-5 рекомендаций | Мера плотности кластера: рекомендации должны быть близки к запросу |
| ILD@5 (Intra-List Diversity) | Среднее попарное расстояние 1 − sim внутри списка | Разнообразие выдачи: список не должен состоять из клонов одного тура |
| Personalization@5 | 1 − среднее попарное пересечение списков для разных запросов | Разным запросам — разные ответы; защита от тривиального решения |

### 1.2 Источник и описание данных

Источник: синтетический датасет data/tours.json, сгенерированный с реалистичной
бизнес-логикой. Цены коррелируют с городом, звёздностью отеля и сезонностью; категории
распределены неравномерно по городам (Мальдивы — только luxury; Барселона и Стамбул —
преимущественно comfort). Текстовые описания туров сгенерированы языковой моделью Claude
(Anthropic) [1]. Персональных и чувствительных данных датасет не содержит.

Структура датасета: 200 туров, 16 полей, файл data/tours.json. Ключевые признаки
описаны в таблице 1.2.

**Таблица 1.2 — Ключевые признаки датасета**

| Признак | Тип | Значения / диапазон |
|---|---|---|

| category | категориальный | luxury (~50 %), comfort, budget |
| meal_type | категориальный | breakfast (~42 %), all_inclusive (~32 %), none (~27 %) |

### 1.3 Разведочный анализ данных

Разведочный анализ проведён в ноутбуке notebooks/exp01_eda.ipynb;

**Шаги предобработки:**

5. TF-IDF-векторизация поля description (max_features=200, ngram_range=(1, 2)) → 200 колонок.

## 2 МОДЕЛИ И ЭКСПЕРИМЕНТЫ

### 2.1 Базовые модели

В качестве quality baseline выбрана **популярность в городе**: для запрошенного тура
возвращаются туры того же города, отсортированные по убыванию звёздности отеля.
Логика: «если ищешь тур в Барселону — покажи лучшие отели Барселоны».

Реализация: функция get_popularity_recs в src/models/recommender.py.

Слабые места baseline:
- Coverage = 0,400 — 60 % каталога никогда не попадает в рекомендации.
- CHR@5 = 0,531 — только половина рекомендаций той же категории, что и запрос
  (категория не учитывается при сортировке).
- Одинаковый список для всех туров одного города.

### 2.2 Улучшенные модели и эксперименты

**Модель 2 — Content-based: структурные признаки.**

Матрица схожести вычисляется как попарная косинусная схожесть структурной матрицы
(one-hot + нормированные числовые, итого 19 признаков).
Два тура считаются схожими, если у них одинаковая категория, город, тип питания
и близкие цена, звёздность, длительность.

Реализация: функции build_feature_matrices (src/models/features.py) и
get_similar_tours (src/models/recommender.py).

Результат: CHR@5 = 0,934, Coverage = 0,982. Структурные признаки хорошо дискриминируют
категорию, но Coverage < 1,0 — несколько туров никогда не рекомендуются.

**Модель 3 — Content-based: TF-IDF по описаниям.**

Матрица схожести вычисляется по косинусной схожести TF-IDF-векторов текстовых описаний
(max_features=200, ngram_range=(1, 2)). Захватывает семантическую близость: туры
с похожими тематическими описаниями («пляжный отдых», «исторический центр»,
«деловой район») получают высокую схожесть.

Результат: Coverage = 1,000 — каждый тур хотя бы раз попадает в рекомендации.
Однако CHR@5 = 0,725 — хуже структурного подхода: описания менее дискриминативны
для категории (luxury / comfort / budget).

**Модель 4 — Комбинированная (финальная).**

Взвешенная сумма двух матриц схожести:

    sim_combined = w_struct · sim_struct + w_text · sim_tfidf

Объединяет точность структурных признаков в категориальной принадлежности
с покрытием TF-IDF. Подбор весов описан в разделе 2.3.

### 2.3 Экспериментальный протокол и сравнение моделей

**Протокол:**
- Данные: 170 активных туров из data/tours.json.
- Разбивка train/val/test не требуется — задача unsupervised retrieval без целевых меток.
  Метрики считаются по всем 170 запросам: каждый тур одновременно является запросом
  и элементом каталога (запрашиваемый тур исключается из рекомендаций).
- Значение top_k = 5 зафиксировано для всех экспериментов.
- Артефакты (матрицы схожести, scaler, векторайзер) сохраняются в artifacts/exp02/
  и artifacts/exp03/ для воспроизводимости.

### 2.4 Выбор финальной модели

Финальная конфигурация: **w_struct = 0,7, w_text = 0,3** (строка 3 таблицы 2.2).

Обоснование:

1. **Coverage = 1,000** — каждый тур хотя бы раз попадает в рекомендации. При w_struct = 0,8
   и 0,9 Coverage опускается до 0,994: часть каталога никогда не показывается пользователям —
   неприемлемо для витрины.
2. **CHR@5 = 0,936** — практически равен максимуму (0,940 при w=0,9), разница в 0,4 %
   статистически незначима.
3. **ILD@5 = 0,355** — умеренное разнообразие: рекомендации не копируют запрос,
   но остаются тематически близкими (при w=0,9 ILD@5 = 0,241 — список однообразен).
4. Конфигурация зафиксирована в artifacts/exp03/best_config.json и configs/config.yaml.

Trade-off: более высокий w_text улучшает ILD, но значительно снижает CHR — главный
бизнес-показатель. Более высокий w_struct максимизирует CHR, но жертвует Coverage
и разнообразием.

## 3 АРХИТЕКТУРА РЕШЕНИЯ И СЕРВИС

### 3.1 Архитектура пайплайна

Пайплайн состоит из двух фаз: offline (подготовка артефактов) и online (сервис).

**Offline (один раз, результат — артефакты):**

    data/tours.json
        └── фильтрация status == "active" (170 туров)
                ├── build_feature_matrices()          # src/models/features.py
                │     ├── one-hot: city_id, category, meal_type  → 16 колонок
                │     ├── MinMaxScaler: hotel_stars, duration_nights, log(price) → 3 колонки
                │     └── TF-IDF: description (max_features=200, ngram=(1,2)) → 200 колонок
                ├── cosine_similarity(struct_matrix) → sim_struct  [170×170]
                ├── cosine_similarity(tfidf_matrix)  → sim_tfidf   [170×170]
                └── 0,7·sim_struct + 0,3·sim_tfidf  → sim_combined [170×170]
                        Сохраняется: artifacts/exp03/sim_combined.npy,
                                     scaler.pkl, tfidf_vectorizer.pkl, tours_active.json

**Online (FastAPI, src/service/app.py):**

    Startup → загрузка sim_combined.npy + tours_active.json в оперативную память
    GET /health  → статус сервиса (статус, число туров, размер матрицы)
    GET /tours   → каталог с фильтрацией по ?city и ?category
    POST /predict → get_similar_tours() → Top-K по sim_combined → ответ за 1–5 мс

### 3.2 API и эндпоинты

Реализованы три эндпоинта. Интерактивная документация доступна по адресу
http://localhost:8000/docs (Swagger UI).

**GET /health** — проверка состояния сервиса:

```
curl http://localhost:8000/health
→ {"status": "ok", "tours_loaded": 170, "sim_matrix_shape": [170, 170]}
```

**GET /tours** — список туров с опциональными фильтрами city и category:

```
curl "http://localhost:8000/tours?city=Istanbul&category=comfort"
```

**POST /predict** — Top-K похожих туров по tour_id:

```
curl -X POST http://localhost:8000/predict \
     -H "Content-Type: application/json" \
     -d '{"tour_id": "362b3800-ccd0-47d2-b8cc-158bd0d22742", "top_k": 5}'

→ {
    "tour_id": "362b3800-ccd0-47d2-b8cc-158bd0d22742",
    "recommendations": [
      {
        "id": "...",
        "name": "Comfort Istanbul — 10 nights",
        "city_name": "Istanbul",
        "category": "comfort",
        "hotel_stars": 3,
        "meal_type": "breakfast",
        "price": 1234.56,
        "similarity": 0.981
      }
    ],
    "elapsed_ms": 1.4
  }
```

Параметр top_k ограничен диапазоном [1, 20] на стороне сервиса. При обращении
к несуществующему tour_id сервис возвращает HTTP 404 с описательным сообщением.

## 4 НАБЛЮДАЕМОСТЬ, КОНФИГУРАЦИЯ И БЕЗОПАСНОСТЬ

### 4.1 Логирование и наблюдаемость

Используется стандартный модуль logging Python. Уровень логирования задаётся через
configs/config.yaml (по умолчанию INFO) и читается при старте сервиса.

Что логируется:

- **Startup:** путь к конфигу, директория артефактов, количество загруженных туров,
  размер матрицы схожести, значение top_k по умолчанию.
- **Каждый запрос /predict:** tour_id, top_k, количество рекомендаций, время ответа в мс.
- **Ошибки:** WARNING при обращении к несуществующему tour_id.

Пример вывода лога:

```
2026-06-17 10:00:01  INFO  Config: /app/configs/config.yaml
2026-06-17 10:00:01  INFO  Loading artifacts from /app/artifacts/exp03
2026-06-17 10:00:01  INFO  Loaded 170 tours, sim_matrix (170, 170), default top_k=5
2026-06-17 10:00:12  INFO  predict tour_id=362b3800-... top_k=5 → 5 recs in 1.4ms
2026-06-17 10:00:15  WARNING  Unknown tour_id=bad-uuid
```

Эндпоинт GET /health позволяет мониторинговой системе убедиться, что сервис поднялся
и загрузил данные:

```json
{"status": "ok", "tours_loaded": 170, "sim_matrix_shape": [170, 170]}
```

### 4.2 Конфигурация

Все параметры, которые могут меняться без изменения кода, вынесены в configs/config.yaml:

```yaml
artifacts_dir: artifacts/exp03     # директория с артефактами модели

model:
  struct_weight: 0.7               # вес структурных признаков
  tfidf_weight: 0.3                # вес TF-IDF описаний
  top_k: 5                         # Top-K по умолчанию
  tfidf_max_features: 200
  tfidf_ngram_range: [1, 2]

service:
  port: 8000
  log_level: INFO
```

Для переключения на результаты другого эксперимента достаточно изменить artifacts_dir
и перезапустить сервис — без изменения исходного кода.

Путь к конфигурационному файлу переопределяется через переменную окружения CONFIG_PATH,
что позволяет использовать разные конфигурации в dev/prod без изменения образа.

### 4.3 Работа с секретами и данными

- Репозиторий не содержит реальных секретов или API-ключей: сервис не обращается
  к внешним API и не требует аутентификации.
- Данные полностью синтетические — персональных данных пользователей нет.
- Файл configs/.env.example документирует единственную переменную окружения
  (CONFIG_PATH); реальный .env включён в .gitignore.
- Параметр top_k ограничен диапазоном [1, 20] для предотвращения злоупотреблений.
- Токены, пароли и PII в логах отсутствуют: логируются только технические
  идентификаторы (tour_id — это UUID без персональных данных).

## ЗАКЛЮЧЕНИЕ

В рамках проекта построен и развёрнут сервис рекомендаций похожих туров
на основе content-based подхода. Реализованы полный ML-пайплайн, REST API и
Docker-контейнеризация.

**Ключевые результаты финальной модели (w_struct = 0,7):**

| Метрика | Значение |
|---|---|
| CHR@5 | 0,936 |
| Coverage | 1,000 |
| MeanSim@5 | 0,720 |
| ILD@5 | 0,355 |
| Personalization@5 | 0,972 |

Комбинированная модель превосходит quality baseline по CHR@5 на 40,5 п.п.
(0,531 → 0,936) и по Coverage на 60 п.п. (0,400 → 1,000).
Задержка эндпоинта /predict составляет 1–5 мс — благодаря хранению матрицы
схожести в оперативной памяти.

**Текущие ограничения:**

- Синтетические данные: перед применением на реальном каталоге потребуется
  переобучение и валидация на пользовательских данных.
- Content-based only: нет учёта поведения пользователей и истории покупок.
- Матрица схожести фиксирована: добавление нового тура требует полного пересчёта
  и перезапуска сервиса.
- Отсутствуют автоматические тесты (папка tests/ содержит только README).

**Направления дальнейшей работы:**

- Гибридная модель: content-based + popularity boost по реальным данным кликов.
- Приближённый поиск ближайших соседей (FAISS, Annoy) для масштабирования
  на каталог из сотен тысяч туров.
- Инкрементальное обновление матрицы при добавлении нового тура без полного перестроения.
- Покрытие кода модульными тестами (pytest) для functions build_feature_matrices,
  get_similar_tours и всех метрик качества.
- Мониторинг дрейфа распределения запросов (Prometheus / встроенные метрики FastAPI).

**Сценарий демонстрации на защите:**

1. Запустить сервис: uv run uvicorn src.service.app:app --host 0.0.0.0 --port 8000
2. Открыть Swagger UI: http://localhost:8000/docs
3. Выполнить GET /health → убедиться, что 170 туров загружено.
4. Выполнить GET /tours?city=Istanbul → скопировать tour_id.
5. Выполнить POST /predict с tour_id и top_k=5 → показать рекомендации
   с полем similarity.
6. Открыть notebooks/exp02_baseline.ipynb — таблица сравнения 4 моделей.
7. Открыть notebooks/exp03_weight_tuning.ipynb — график подбора весов.
